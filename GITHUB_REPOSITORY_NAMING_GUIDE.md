## 🎨 Naming Convention

### Primary Pattern: `[Technology/Domain]-[Function]`

**Examples:**
- `webgpu-profiler` (Technology + Function)
- `vector-search` (Domain + Function)
- `cascade-router` (Unique Branding + Function)

### Naming Rules

1. **Use kebab-case** (all lowercase, hyphens between words)
2. **Lead with strongest keyword** (webgpu, gpu, ai, ml)
3. **Keep it short** (2-3 words maximum)
4. **Avoid redundancy** (no "js", "lib", "tool" suffixes unless critical)
5. **Be descriptive** (name should indicate what the tool does)
6. **Think SEO** (include high-volume search terms)

### What to Avoid ❌

- ❌ `my-awesome-tool` (subjective, not searchable)
- ❌ `tool-name-js` (redundant "js" suffix)
- ❌ `superinstance-tool-name` (org prefix in repo name is redundant)
- ❌ `the-tool-name` (unnecessary article)
- ❌ `toolname-v2` (versioning in repo name)
- ❌ `tool-name-library` (redundant "library")
- ❌ `tool-name-framework` (unless it truly is a framework)

## 🔄 Migration Strategy

### **Repository Renaming Process**

**1. Preparation:**
```bash
# Update package.json
cd packages/old-name
npm pkg set name="@superinstance/new-name"

# Update all imports
find . -type f -name "*.ts" -exec sed -i 's/old-name/new-name/g' {} +

# Update README links
sed -i 's/old-name/new-name/g' README.md
```

**2. GitHub Rename:**
- Go to repository Settings
- Change repository name
- GitHub automatically redirects old URLs

**3. Publish to NPM:**
```bash
# Publish new package name
npm publish --access public

# Deprecate old package (optional)
npm deprecate @superinstance/old-name "Use @superinstance/new-name instead"
```

**4. Update Documentation:**
- Update all README.md files
- Update integration examples
- Update getting started guides
- Update API documentation

### **Safe Transition Period**

- Keep both package names working for 6 months
- Add deprecation notice to old package
- Update all examples to use new names
- Create migration guide for users

## 🎓 Decision Framework

### **When Choosing a Repository Name, Ask:**

1. **Is it searchable?** (Will people search for this?)
2. **Is it memorable?** (Can people remember it after hearing it once?)
3. **Is it clear?** (Does the name indicate what it does?)
4. **Is it short?** (Is it 2-3 words or less?)
5. **Is it consistent?** (Does it follow our naming convention?)
6. **Is it available?** (Is the name free on GitHub and npm?)
7. **Is it professional?** (Does it sound production-ready?)

### **Scoring System:**

Rate each potential name 1-5 on:
- SEO Value (search volume)
- Clarity (how clear the purpose is)
- Memorability (how easy to remember)
- Brevity (how short it is)
- Consistency (how well it fits our pattern)

**Target Score:** 20+ out of 25
