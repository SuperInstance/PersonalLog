**1. Create Notes**
- Click "New Note" button

  ```markdown
  # My Project Ideas
  ```

- Click "Use as Context" button
- Send note content to chatbot

## 🧠 RAG & Memory System Guide

### Understanding the Memory System

**User Memory (Cross-Project Patterns)**
- **Categories**: style, technique, preference, workflow
- **Source**: human-editable or AI-learned
- **Tracking**: confidence, usage count, success rate, last used
- **Examples**:
  - "preferred_model": "SDXL"
  - "color_scheme": "warm_colors"
  - "lighting": "dramatic"

**Project Memory (Project-Specific Learning)**
- **Categories**: successful_prompt, user_preference, effective_technique
- **Source**: conversation, iteration, feedback, ai_analysis
- **Tracking**: importance, confidence, user rating, use count
- **Examples**:
  - "successful_prompt": "8k resolution, detailed textures"
  - "user_preference": "prefers warm lighting"
  - "effective_technique": "ControlNet for poses"

**Cross-Project References**
- **Types**: style_influence, technique_adoption, element_reuse
- **Strength**: 0-1 (how strong the reference is)
- **Tracking**: use count, last used
- **Examples**:
  - "style_influence": "Use neon lighting from CyberpunkCityscape project"
  - "technique_adoption": "Use prompt technique from FantasyLandscape project"

### Using the Memory Editor

1. **Open Memory Editor Panel**
   - Tab: "Memory Editor"
   - View all cross-project patterns
   - Edit human-readable patterns

2. **Edit User Memory**
   - Click pattern to edit
   - Modify key, value, or category
   - Update confidence (0-1)
   - Add notes or description
   - Click "Save" to update

3. **Reset Memory**
   - Click "Reset" button for a pattern
   - Clear all user memory
   - Start fresh with automatic learning

**Learning Process:**

## 🛡️ Production Deployment Guide

### Option 1: Vercel (Recommended) ⭐

**Why Vercel?**
- Zero configuration required
- Automatic SSL certificates
- Global CDN for static assets
- Automatic scaling with edge network
- Preview deployments for testing
- Production deployment in one command
- Environment variables in dashboard

**Steps:**
```bash
# 1. Install Vercel CLI
bun install -g vercel

# 2. Login to Vercel
vercel login

# 3. Deploy to production
vercel --prod

# 4. Set environment variables in Vercel dashboard
# - DATABASE_URL
# - ZAI_API_KEY
# - SECRET_KEY
# - NODE_ENV=production
```

**Vercel Features:**
- Automatic builds from Git
- Edge network for fast responses
- Serverless functions (API routes)
- File system storage (for uploads)
- Automatic SSL
- Custom domain support
- Preview deployments (PR = auto-deploy)

**Environment Variables:**
Set these in Vercel dashboard:
- `NODE_ENV`: `production`
- `DATABASE_URL`: Your database connection string
- `ZAI_API_KEY`: Your Z.ai API key
- `SECRET_KEY`: Generate a secure random string (min 32 chars)
- `ALLOWED_ORIGINS`: `https://yourdomain.com,https://www.yourdomain.com`

### Option 2: Docker Deployment

**Why Docker?**
- Full control over environment
- Consistent across deployments
- Easy scaling with containers
- Orchestration support (Kubernetes, Docker Swarm)
- Production-grade containerization

**Steps:**
```bash
# 1. Build Docker image
docker build -t comfyui-vibe-agent .

# 2. Run container locally (for testing)
docker run -p 3000:3000 \
  -e DATABASE_URL=file:./db/custom.db \
  -e ZAI_API_KEY=${ZAI_API_KEY} \
  comfyui-vibe-agent

# 3. Use docker-compose for multi-service setup
docker-compose up -d

# 4. Deploy to container registry
docker push comfyui-vibe-agent
```

**Docker Features:**
- Self-contained environment
- Reproducible builds
- Easy scaling (horizontal)
- Service discovery (with orchestration)
- Volume mounting for data persistence
- Health checks with automatic restarts

**Docker Compose:**
```yaml
version: '3.8'
services:
  app:
    build: .
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=production
      - DATABASE_URL=file:./db/custom.db
      - ZAI_API_KEY=${ZAI_API_KEY}
      - SECRET_KEY=${SECRET_KEY}
    volumes:
      - ./db:/app/db
      - ./uploads:/app/uploads
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3000/api/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s
```

## 🐛 Troubleshooting

### Common Production Issues

**Issue: Build fails with TypeScript errors**
```
Solution:
1. bunx tsc --noEmit  # Check for type errors
2. Fix all type errors
3. Run bunx run build again
```

**Issue: API returns 429 Rate Limit Exceeded**
```
Solution:
1. Wait for rate limit to reset (check Retry-After header)
2. Reduce request frequency
3. Increase rate limit in api-middleware.ts if needed
4. Check logs for legitimate use vs abuse
```

**Issue: File upload fails with validation error**
```
Solution:
1. Check file size against limits (50MB for images, 100MB for audio)
2. Verify file type is allowed
3. Check file MIME type matches actual content
4. Remove any non-standard characters from file name
5. Try smaller file size
```

**Issue: Health check returns unhealthy status**
```
Solution:
1. Check database connection: ps aux | grep sqlite
2. Check API service: Verify server is running
3. Check rate limiter: Review logs for abuse
4. Verify all dependencies are installed: bun install
5. Check environment variables: Verify all required vars are set
6. Restart services: bunx start
```

**Issue: Performance is slow**
```
Solution:
1. Check response times in analytics dashboard
2. Add database indexes: Add indexes to prisma/schema.prisma
3. Optimize queries: Use select, avoid N+1 queries
4. Enable caching: Implement Redis or use Next.js caching
5. Scale horizontally: Add more servers or containers
6. Review and optimize heavy operations
```

**Issue: STT not working**
```
Solution:
1. Check ZAI_API_KEY is set correctly
2. Verify microphone permissions in browser
3. Test with supported audio format (WAV, MP3)
4. Check browser console for errors
5. Verify ASR skill is available in z-ai-web-dev-sdk
6. Test with shorter audio clips (1-2 minutes)
```
